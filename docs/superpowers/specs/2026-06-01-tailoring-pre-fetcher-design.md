# Design: tailoring.xml `%pre` fetcher

**Date:** 2026-06-01
**Status:** approved, queued for implementation
**Companion plan:** `docs/superpowers/plans/2026-06-01-tailoring-pre-fetcher-implementation.md` (to be written)

## Problem

The kickstart generated by `ks-gen gen` configures `oscap-anaconda-addon`
with `tailoring-path = /tailoring.xml`. That path is interpreted by the
addon as a **literal path inside the installer's runtime filesystem**, not
a URL. The two delivery paths `ks-gen` documents today both fail to
satisfy that requirement:

- **HTTP delivery** (`MINIMAL-TEST.md` walkthrough): the operator serves
  the four-file bundle over `python -m http.server`. Anaconda fetches
  `ks.cfg` but never fetches `tailoring.xml` — the addon only looks at
  `/tailoring.xml` in the installer runtime, which is empty.
- **ISO delivery** (`ks-gen iso`): `xorriso` injects `tailoring.xml` at
  the ISO root, but the ISO is mounted at `/run/install/repo` during
  install, not at `/`. The addon still finds nothing at `/tailoring.xml`.

The symptom in both cases is silent: the addon falls back to the
unmodified base STIG profile. That profile is exactly what `ks-gen`
exists to neutralize — its `ssh_keep_open`, `admin_user_and_keys`, and
`faillock_safety` overrides are what keep the operator from locking
themselves out. A "successful" install that silently skipped tailoring
is the worst possible outcome: it looks fine, then bricks remote access.

Discovered during the v0.1.0 acceptance test on 2026-06-01: Anaconda's
HTTP fetches showed only `GET /ks.cfg`, never `GET /tailoring.xml`,
despite `MINIMAL-TEST.md` Step 5 promising both.

## Goals

- HTTP-served bundles deliver `tailoring.xml` to the addon's expected
  path before `%addon` runs.
- `ks-gen iso` bundles do the same, by copying from the install media
  Anaconda has already mounted.
- A failure to fetch / copy hard-fails the install rather than silently
  applying base STIG.
- No new Python modules; the fix is a small `%pre` block in the existing
  `ks.cfg` Jinja template.

## Non-goals

- Supporting `nfs:`, `cdrom:`, or other `inst.ks=` transports. v0.1
  ships HTTP and `hd:LABEL` only; other transports remain unsupported
  in v0.1.1 and the `%pre` block exits non-zero on them with a clear
  message.
- Rewriting `isolinux`/`grub` boot configs inside `ks-gen iso` so the
  operator no longer has to type `inst.ks=...` at the boot prompt.
  Tracked separately for v0.2.
- Any shellcheck / docker-based runtime test of the `%pre` body —
  Anaconda's `%pre` runs inside a busybox-ish stage 2 environment that
  is high-effort to reproduce in CI. The Hyper-V manual acceptance test
  in `MINIMAL-TEST.md` is the integration-level verification.

## Design

### Where the fetch lives

A single `%pre` block is emitted unconditionally at the top of every
generated `ks.cfg`, immediately before `%packages`. There is no
configuration knob — every bundle needs tailoring, regardless of how
the operator chooses to deliver it.

Anaconda parses the entire kickstart file before executing any phase
and runs `%pre` before `%addon`, so placing the block in the template
guarantees `/tailoring.xml` is in place by the time the oscap addon
initializes.

### How it derives the source

The block reads `/proc/cmdline` to extract the `inst.ks=` argument the
operator supplied at the GRUB prompt (HTTP case) or that the boot media
configured (ISO case). It then branches on the scheme:

- `http://*` or `https://*` — derive the base URL by stripping the last
  path component (`/ks.cfg`), then `curl` `${base}/tailoring.xml` to
  `/tailoring.xml`.
- `hd:*` — copy from `/run/install/repo/tailoring.xml` (the path
  Anaconda mounts kickstart media at) to `/tailoring.xml`.
- anything else — print a clear error and exit non-zero, aborting the
  install.

### Failure handling

`%pre --erroronfail` means a non-zero exit aborts the install with a
loud error. The block uses `set -euo pipefail` so unexpected shell
errors surface the same way.

Within the recognized branches:

- HTTP uses `curl -fsSL --retry 5 --retry-delay 3`. Five attempts at
  three-second intervals covers transient network blips without
  requiring intervention.
- `hd:` exits with a clear error if `cp` fails. `cp -f` failures from a
  mounted ISO are not transient (ENOENT / permission), so retrying the
  same command would recover nothing; the explicit fail-loud matches the
  HTTP branch (which relies on `curl -fsSL` exiting under `set -e`).

After the fetch / copy, the block verifies `/tailoring.xml` is non-empty
and that its first bytes are `<?xml`, as a cheap sanity check against
truncation or 404s served as 200s by a misconfigured server.

### Idempotence

The block short-circuits with `exit 0` if `/tailoring.xml` is already
present and non-empty. This is for two cases:

- Operators who pre-stage `tailoring.xml` via a custom `inst.dd` or
  `inst.stage2` mechanism continue to work.
- The block is safe to re-run (Anaconda doesn't, but future contributors
  inspecting the script shouldn't need to worry about ordering).

### The block, verbatim

```bash
%pre --erroronfail --log=/tmp/ks-pre-tailoring.log
set -euo pipefail

if [ -s /tailoring.xml ]; then
  echo "ks-gen: /tailoring.xml already present, skipping fetch"
  exit 0
fi

ks_arg=$(awk -F'inst.ks=' 'NF>1{print $2}' /proc/cmdline | awk '{print $1}')
case "$ks_arg" in
  http://*|https://*)
    base="${ks_arg%/*}"
    curl -fsSL --retry 5 --retry-delay 3 "${base}/tailoring.xml" -o /tailoring.xml
    ;;
  hd:*)
    cp -f /run/install/repo/tailoring.xml /tailoring.xml || {
      echo "ks-gen: failed to copy tailoring.xml from /run/install/repo" >&2
      exit 1
    }
    ;;
  *)
    echo "ks-gen: unsupported inst.ks transport '$ks_arg'; cannot fetch tailoring.xml" >&2
    exit 1
    ;;
esac

test -s /tailoring.xml
head -c 5 /tailoring.xml | grep -q '<?xml'
%end
```

## Testing strategy

| Layer | What it asserts | Mechanism |
|---|---|---|
| Snapshot | All four golden scenarios contain the new `%pre` block byte-for-byte | Regenerate `tests/golden/snapshots/{minimal-dhcp,stig-strict,modern-crypto,bare-metal-usbguard}/ks.cfg` |
| Invariant | The block appears once, before `%addon`, mentions both transport branches, uses `--erroronfail`, ends with the `<?xml` sentinel | New parametrized test in `tests/test_skeleton.py` |
| Idempotence | The `[ -s /tailoring.xml ]` guard is present | Same invariant test — static, since `%pre` can't run in pytest |
| End-to-end | Hyper-V VM successfully completes install, ssh works, `oscap` deviations match `exceptions.md` | Manual `MINIMAL-TEST.md` walkthrough |

## Branch and release strategy

The fix lands as additional commits on `impl/v0.1.0`. PR #1 picks them
up automatically. No version bump — v0.1.0 has not yet been tagged, so
the bug never shipped under a release name. The PR's "Manual install
verification" test-plan box gets ticked once the re-run of
`MINIMAL-TEST.md` passes end to end.

## Open questions

None at design time. Behavior for unrecognized `inst.ks=` transports is
explicitly hard-fail; if future work needs to add `nfs:` or `cdrom:`
support, it's a new branch in the `case` statement and a docs update.
