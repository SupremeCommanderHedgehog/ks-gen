# Design: tailoring.xml `%pre` fetcher

**Date:** 2026-06-01
**Status:** SUPERSEDED on 2026-06-01 — the `%pre` + `%addon` approach
documented below failed on the AlmaLinux 9 acceptance test. See the
**Superseded by** section at the bottom for what shipped instead.
**Companion plan:** `docs/superpowers/plans/2026-06-01-tailoring-pre-fetcher-implementation.md`

## Problem

The kickstart generated by `ks-gen gen` configures `oscap-anaconda-addon`
with `tailoring-path = /tailoring.xml`. The addon **resolves that path
inside its content staging directory `/tmp/openscap_data/`**, not at the
installer-FS root. The two delivery paths `ks-gen` documents today both
fail to satisfy that requirement:

- **HTTP delivery** (`MINIMAL-TEST.md` walkthrough): the operator serves
  the four-file bundle over `python -m http.server`. Anaconda fetches
  `ks.cfg` but no mechanism places `tailoring.xml` inside
  `/tmp/openscap_data/`. The addon aborts with
  `expected a file /tmp/openscap_data/tailoring.xml to be part of the
  supplied content`.
- **ISO delivery** (`ks-gen iso`): `xorriso` injects `tailoring.xml` at
  the ISO root, but the ISO is mounted at `/run/install/repo` during
  install. The addon's content directory is independent of the ISO mount
  and the same lookup miss occurs.

When the addon aborts, the install fails loudly — but a slightly older
addon version, or any failure path that produces a soft fallback, would
quietly apply the unmodified base STIG profile instead. That profile is
exactly what `ks-gen` exists to neutralize — its `ssh_keep_open`,
`admin_user_and_keys`, and `faillock_safety` overrides are what keep the
operator from locking themselves out.

Discovery timeline (2026-06-01): the original v0.1.0 acceptance test
showed only `GET /ks.cfg`, never `GET /tailoring.xml`, despite
`MINIMAL-TEST.md` Step 5 promising both. An initial fix added a `%pre`
block that wrote to `/tailoring.xml` in the installer runtime — the
re-run produced both GETs but Anaconda then aborted with the
`/tmp/openscap_data/tailoring.xml` error above. This spec reflects the
corrected target path; the original (wrong) target path lived only in
intermediate commits on `impl/v0.1.0`, never in a tagged release.

## Goals

- HTTP-served bundles deliver `tailoring.xml` to the addon's expected
  path before `%addon` runs.
- `ks-gen iso` bundles do the same, by copying from the install media
  Anaconda has already mounted.
- A failure to fetch / copy hard-fails the install rather than silently
  applying base STIG.
- No new Python modules; the fix is a small `%pre` block in the existing
  `ks.cfg` Jinja template.

## Non-goals

- Supporting `nfs:`, `cdrom:`, or other `inst.ks=` transports. v0.1
  ships HTTP and `hd:LABEL` only; other transports remain unsupported
  in v0.1.1 and the `%pre` block exits non-zero on them with a clear
  message.
- Rewriting `isolinux`/`grub` boot configs inside `ks-gen iso` so the
  operator no longer has to type `inst.ks=...` at the boot prompt.
  Tracked separately for v0.2.
- Any shellcheck / docker-based runtime test of the `%pre` body —
  Anaconda's `%pre` runs inside a busybox-ish stage 2 environment that
  is high-effort to reproduce in CI. The Hyper-V manual acceptance test
  in `MINIMAL-TEST.md` is the integration-level verification.

## Design

### Where the fetch lives

A single `%pre` block is emitted unconditionally at the top of every
generated `ks.cfg`, immediately before `%packages`. There is no
configuration knob — every bundle needs tailoring, regardless of how
the operator chooses to deliver it.

Anaconda parses the entire kickstart file before executing any phase
and runs `%pre` before `%addon`, so placing the block in the template
guarantees `/tmp/openscap_data/tailoring.xml` is in place by the time
the oscap addon initializes. The block `mkdir -p`s
`/tmp/openscap_data/` first — the directory does not yet exist when
`%pre` runs; the addon would create it itself later, but we need it
earlier.

### How it derives the source

The block reads `/proc/cmdline` to extract the `inst.ks=` argument the
operator supplied at the GRUB prompt (HTTP case) or that the boot media
configured (ISO case). It then branches on the scheme:

- `http://*` or `https://*` — derive the base URL by stripping the last
  path component (`/ks.cfg`), then `curl` `${base}/tailoring.xml` to
  `/tmp/openscap_data/tailoring.xml`.
- `hd:*` — copy from `/run/install/repo/tailoring.xml` (the path
  Anaconda mounts kickstart media at) to `/tmp/openscap_data/tailoring.xml`.
- anything else — print a clear error and exit non-zero, aborting the
  install.

### Failure handling

`%pre --erroronfail` means a non-zero exit aborts the install with a
loud error. The block uses `set -euo pipefail` so unexpected shell
errors surface the same way.

Within the recognized branches:

- HTTP uses `curl -fsSL --retry 5 --retry-delay 3`. Five attempts at
  three-second intervals covers transient network blips without
  requiring intervention.
- `hd:` exits with a clear error if `cp` fails. `cp -f` failures from a
  mounted ISO are not transient (ENOENT / permission), so retrying the
  same command would recover nothing; the explicit fail-loud matches the
  HTTP branch (which relies on `curl -fsSL` exiting under `set -e`).

After the fetch / copy, the block verifies
`/tmp/openscap_data/tailoring.xml` is non-empty and that its first
bytes are `<?xml`, as a cheap sanity check against truncation or 404s
served as 200s by a misconfigured server.

### Idempotence

The block short-circuits with `exit 0` if
`/tmp/openscap_data/tailoring.xml` is already present and non-empty.
This is for two cases:

- Operators who pre-stage `tailoring.xml` via a custom `inst.dd` or
  `inst.stage2` mechanism continue to work.
- The block is safe to re-run (Anaconda doesn't, but future contributors
  inspecting the script shouldn't need to worry about ordering).

### The block, verbatim

```bash
%pre --erroronfail --log=/tmp/ks-pre-tailoring.log
set -euo pipefail

# oscap-anaconda-addon resolves tailoring-path inside its content staging dir
# /tmp/openscap_data, not the installer-FS root. Create it and write here.
mkdir -p /tmp/openscap_data

if [ -s /tmp/openscap_data/tailoring.xml ]; then
  echo "ks-gen: /tmp/openscap_data/tailoring.xml already present, skipping fetch"
  exit 0
fi

ks_arg=$(awk -F'inst.ks=' 'NF>1{print $2}' /proc/cmdline | awk '{print $1}')
case "$ks_arg" in
  http://*|https://*)
    base="${ks_arg%/*}"
    curl -fsSL --retry 5 --retry-delay 3 "${base}/tailoring.xml" -o /tmp/openscap_data/tailoring.xml
    ;;
  hd:*)
    cp -f /run/install/repo/tailoring.xml /tmp/openscap_data/tailoring.xml || {
      echo "ks-gen: failed to copy tailoring.xml from /run/install/repo" >&2
      exit 1
    }
    ;;
  *)
    echo "ks-gen: unsupported inst.ks transport '$ks_arg'; cannot fetch tailoring.xml" >&2
    exit 1
    ;;
esac

test -s /tmp/openscap_data/tailoring.xml
head -c 5 /tmp/openscap_data/tailoring.xml | grep -q '<?xml'
%end
```

## Testing strategy

| Layer | What it asserts | Mechanism |
|---|---|---|
| Snapshot | All four golden scenarios contain the new `%pre` block byte-for-byte | Regenerate `tests/golden/snapshots/{minimal-dhcp,stig-strict,modern-crypto,bare-metal-usbguard}/ks.cfg` |
| Invariant | The block appears once, before `%addon`, mentions both transport branches, uses `--erroronfail`, ends with the `<?xml` sentinel | New parametrized test in `tests/test_skeleton.py` |
| Idempotence | The `[ -s /tmp/openscap_data/tailoring.xml ]` guard is present | Same invariant test — static, since `%pre` can't run in pytest |
| End-to-end | Hyper-V VM successfully completes install, ssh works, `oscap` deviations match `exceptions.md` | Manual `MINIMAL-TEST.md` walkthrough |

## Branch and release strategy

The fix lands as additional commits on `impl/v0.1.0`. PR #1 picks them
up automatically. No version bump — v0.1.0 has not yet been tagged, so
the bug never shipped under a release name. The PR's "Manual install
verification" test-plan box gets ticked once the re-run of
`MINIMAL-TEST.md` passes end to end.

## Open questions

None at design time. Behavior for unrecognized `inst.ks=` transports is
explicitly hard-fail; if future work needs to add `nfs:` or `cdrom:`
support, it's a new branch in the `case` statement and a docs update.

## Superseded by

This design lived for ~90 minutes on 2026-06-01 before the AlmaLinux 9
Hyper-V acceptance test killed it. Two failure modes surfaced in
sequence:

1. **First `%pre` revision** wrote `tailoring.xml` to `/tailoring.xml`
   in the installer runtime FS. Anaconda's `oscap-anaconda-addon`
   resolves `tailoring-path` inside its content staging directory
   (`/tmp/openscap_data/`), so the file was looked for at
   `/tmp/openscap_data/tailoring.xml` and not found. Anaconda dropped
   to the rescue shell.
2. **Second `%pre` revision** correctly wrote to
   `/tmp/openscap_data/tailoring.xml` plus `mkdir -p`d the directory.
   The addon still refused, this time with:

       expected a file /tmp/openscap_data/tailoring.xml to be part of
       the supplied content, but it was not the case, got only
       ['/usr/share/xml/scap/ssg/content//ssg-almalinux9-ds.xml']

   Reading the upstream addon source revealed that "supplied content"
   isn't a directory listing — it's a list of files the addon's own
   content-handling pipeline registered. For `content-type = scap-
   security-guide`, only the SSG datastream is registered. Tailoring
   files have to be inside an archive the addon downloaded itself
   (`content-type = archive` + `content-url = ...`). A static `%pre`
   that writes to disk cannot make the addon's check pass.

The architecture was reworked to drop the addon entirely. The kickstart
now emits a leading `%post` block that `curl`s `tailoring.xml` from the
same base URL `inst.ks=` used and runs `oscap xccdf eval --remediate
--tailoring-file /root/tailoring.xml ...` directly — which is what the
addon does internally anyway. Side benefits: `tailoring.xml`, the
remediation ARF, and the HTML report all land in `/root/` on the
installed system for later audit.

The plan file
(`docs/superpowers/plans/2026-06-01-tailoring-pre-fetcher-implementation.md`)
is preserved as the historical record of the abandoned approach; do not
follow it for new work. For the architecture as it actually shipped,
see §2.1 of `2026-06-01-alma-stig-kickstart-design.md`.
