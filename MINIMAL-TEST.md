# MINIMAL-TEST.md — end-to-end install verification on Hyper-V

This walks through installing the `minimal-dhcp` golden scenario in a Hyper-V
VM, then verifying the resulting AlmaLinux 9 system is STIG-compliant and
remote-reachable. Total wall-clock: ~20 minutes once Hyper-V is up.

Use this as the v0.1.0 acceptance test before merging
[PR #1](https://github.com/SupremeCommanderHedgehog/ks-gen/pull/1).

## Prerequisites

- Windows 11 Pro with **Hyper-V installed and the host rebooted at least once
  since enablement**. See the "Installing Hyper-V" appendix at the bottom if
  this isn't done.
- This repo at `C:\Users\yizshachuck\source\alma-linux-security`.
- `AlmaLinux-9-latest-x86_64-dvd.iso` present at the repo root (gitignored).
- `.venv` populated with dev deps:
  `.venv\Scripts\python.exe -m pip install -e ".[dev]"`.
- **Two admin PowerShell windows** — one will host the HTTP server, the other
  will run Hyper-V cmdlets.
- A real SSH keypair to log in with. The golden YAML ships a placeholder
  public key; replace it before running, or use `ks-gen new` to produce a
  custom bundle.

## Step 1 — Generate the bundle (PowerShell window #1)

```powershell
cd C:\Users\yizshachuck\source\alma-linux-security
.venv\Scripts\python.exe -m ks_gen gen `
  --config tests\golden\minimal-dhcp.host.yaml `
  --set disk.preset=minimal `
  --out build\web01
```

`--set disk.preset=minimal` shrinks the partition layout to fit in a 20 GiB
VM disk. The `stig_server` preset (default) needs ~50 GiB and isn't required
for this smoke test.

Confirm the four-file bundle landed:

```powershell
ls build\web01
# host.yaml, ks.cfg, tailoring.xml, exceptions.md
```

## Step 2 — Serve the bundle over HTTP (still window #1)

```powershell
cd build\web01
python -m http.server 8000
```

**Leave this window running** for the rest of the test. You'll see
`GET /ks.cfg` and `GET /tailoring.xml` log lines when Anaconda fetches the
files — that's the "yes, the install found us" signal.

## Step 3 — Note the Default Switch IP (PowerShell window #2, admin)

```powershell
$HOST_IP = (Get-NetIPAddress -InterfaceAlias 'vEthernet (Default Switch)' `
            -AddressFamily IPv4).IPAddress
$HOST_IP
```

Typically a `172.x.x.1` address. Sanity check that the bundle is reachable
from outside the loopback (some firewall configs block this):

```powershell
curl "http://$HOST_IP:8000/ks.cfg" | Select-Object -First 5
```

You should see `# Generated by ks-gen …`, `text`, `lang en_US.UTF-8`, etc. If
this 404s or times out, fix it before booting the VM.

## Step 4 — Create the VM (still window #2)

```powershell
$ISO  = 'C:\Users\yizshachuck\source\alma-linux-security\AlmaLinux-9-latest-x86_64-dvd.iso'
$VHDX = 'C:\Hyper-V\Virtual Hard Disks\ks-gen-web01.vhdx'
New-Item -ItemType Directory -Force -Path (Split-Path $VHDX) | Out-Null

New-VM -Name 'ks-gen-web01' `
       -Generation 2 `
       -MemoryStartupBytes 4GB `
       -NewVHDPath $VHDX -NewVHDSizeBytes 20GB `
       -SwitchName 'Default Switch'

Set-VMProcessor -VMName 'ks-gen-web01' -Count 2
Set-VMFirmware  -VMName 'ks-gen-web01' -EnableSecureBoot Off
Add-VMDvdDrive  -VMName 'ks-gen-web01' -Path $ISO
$dvd = Get-VMDvdDrive -VMName 'ks-gen-web01'
Set-VMFirmware  -VMName 'ks-gen-web01' -FirstBootDevice $dvd
```

Secure Boot is off because AlmaLinux's signed shim isn't trusted by Hyper-V's
default Microsoft UEFI template. For a test install this is fine; production
images need the open-source Secure Boot template instead.

## Step 5 — Boot and inject `inst.ks=` at GRUB

```powershell
vmconnect.exe localhost 'ks-gen-web01'
Start-VM 'ks-gen-web01'
```

When the GRUB menu appears in the vmconnect console:

1. Highlight "Install AlmaLinux 9" (the top entry).
2. Press **`e`** to edit.
3. Find the `linuxefi /images/pxeboot/vmlinuz` line (usually the second-last
   line in the entry).
4. Move the cursor to the end of that line.
5. Append (note the leading space):

   ```
    inst.ks=http://<HOST_IP>:8000/ks.cfg
   ```

   Substitute the IP captured in Step 3.

6. Press **Ctrl-X** to boot.

Within ~30 seconds, window #1 should log Anaconda's kickstart fetch:

```
"GET /ks.cfg HTTP/1.1" 200 -
```

Then the install proceeds unattended — kernel boot, partition, package
install (this is the slow part). Several minutes in, the first `%post`
block fires and you should see a second GET:

```
"GET /tailoring.xml HTTP/1.1" 200 -
```

That second GET is `curl` from inside the chroot, fetching tailoring.xml
from the same base URL the operator's `inst.ks=` pointed at. The block
then runs `oscap xccdf eval --remediate` against the installed system,
followed by a second `%post` that applies ks-gen's rule overrides
(admin user + sshd config + crypto policy + …). If you see only the
first GET and the install stalls, inspect `/root/ks-post-oscap.log` on
the installed system after reboot (it survives the install for audit).

Total wall-clock: ~10-15 minutes on a modern host, then a reboot.

## Step 6 — Find the VM's IP after reboot

```powershell
Get-VM 'ks-gen-web01' | Select-Object -ExpandProperty NetworkAdapters |
  Select-Object IPAddresses
```

Usually a `172.x.x.x` address on the Default Switch's NAT range. If
`IPAddresses` is empty, give it another 30 seconds — chrony may still be
stabilizing the network.

## Step 7 — SSH in

```powershell
ssh opsadmin@<vm-ip>
```

The admin username is `opsadmin` (per `tests/golden/minimal-dhcp.host.yaml`).
Use the private key matching the public key in that YAML. If you didn't
replace the placeholder key in Step 1, **you won't be able to log in** —
regenerate the bundle with your real key (`ks-gen new` walks you through it)
and re-run from Step 4 with a fresh VM.

## Step 8 — Verify STIG compliance and remote safety

Once logged in over SSH:

```bash
# Confirm crypto policy
update-crypto-policies --show
# Expected: DEFAULT (the minimal-dhcp scenario uses crypto.policy=MODERN)

# Confirm sshd config
sudo sshd -T | egrep -i '^(port|permitrootlogin|passwordauth|clientalive)'
# Expected: Port 22, PermitRootLogin no, PasswordAuthentication no,
#           ClientAliveInterval 600, ClientAliveCountMax 1

# Confirm firewalld port
sudo firewall-cmd --list-ports
# Expected: 22/tcp

# Confirm civilian banner — should NOT contain "U.S. Government" or "USG"
cat /etc/issue
sudo cat /etc/issue.net

# Confirm Ed25519 host key exists (MODERN crypto path)
ls /etc/ssh/ssh_host_ed25519_key.pub

# Verify unattended-update timers are scheduled
systemctl list-timers --all | grep -E 'dnf-automatic|reboot-if-needed'
# Expected: three timer entries with future NEXT columns —
#           dnf-automatic.timer, ks-gen-dnf-automatic-full.timer, and
#           ks-gen-reboot-if-needed.timer. If dnf-automatic.timer's schedule
#           shows 06:00 (the SSG default) rather than the YAML-configured
#           nightly 02:00, the drop-in at
#           /etc/systemd/system/dnf-automatic.timer.d/ks-gen.conf wasn't
#           picked up — re-check the %post log at /root/ks-post.log

# tailoring.xml is preserved on the installed FS at /root/tailoring.xml
# by the oscap %post block. The oscap remediation report and ARF results
# from install time also live there.
ls -l /root/tailoring.xml /root/oscap-remediation-*.{xml,html}

# Re-run oscap evaluation against the live system to see what state landed
sudo oscap xccdf eval \
  --profile xccdf_org.ssgproject.content_profile_stig \
  --tailoring-file /root/tailoring.xml \
  /usr/share/xml/scap/ssg/content/ssg-almalinux9-ds.xml | tail -40
```

The oscap output will list failed rules. Every "fail" should correspond to a
row in `build\web01\exceptions.md` on the host. Any oscap fail that **isn't**
in `exceptions.md` is real compliance drift — file it as a bug against
`ks-gen`.

## Cleanup

When done with the test VM:

```powershell
Stop-VM 'ks-gen-web01' -Force
Remove-VM 'ks-gen-web01' -Force
Remove-Item 'C:\Hyper-V\Virtual Hard Disks\ks-gen-web01.vhdx'
```

Then Ctrl-C the HTTP server in window #1.

## Iterating

To test a different scenario (e.g., `stig-strict` with FIPS), the workflow is:

1. Stop/remove the previous VM and VHDX (above).
2. Regenerate the bundle: substitute a different YAML (or `ks-gen new`) and
   adjust the disk size to ≥50 GiB if using `stig_server`.
3. Re-run Steps 2 through 8.

The HTTP server and host IP are reusable across runs — only the VM and
bundle change.

## Appendix — Installing Hyper-V (one-time)

If `Get-VMHost` errors with "not recognized as a cmdlet", Hyper-V isn't
installed or hasn't been rebooted into.

From an admin PowerShell:

```powershell
# The PowerShell cmdlet form (Enable-WindowsOptionalFeature) often fails with
# "Class not registered" under PowerShell 7. Use dism directly — it's the
# same operation without the COM indirection.
dism /online /enable-feature /featurename:Microsoft-Hyper-V-All /all /norestart

# A restart is required for the runtime + management module to load.
Restart-Computer
```

After reboot, sanity-check:

```powershell
Get-WindowsOptionalFeature -Online -FeatureName Microsoft-Hyper-V-All
# State : Enabled

Get-VMHost
# Should print host info, not "not recognized".
```

If `Get-VMHost` still errors:

```powershell
Get-Module -ListAvailable Hyper-V
# Should list a Hyper-V module. If empty, the feature didn't fully install.

Import-Module Hyper-V
# If this errors under PowerShell 7, open Windows PowerShell 5.1
# (`powershell.exe` — the blue console, not pwsh.exe) and retry from there.
```

Prerequisites for Hyper-V to actually run:

- Win11 Pro/Enterprise/Education (not Home).
- VT-x or AMD-V enabled in BIOS/UEFI.
- SLAT-capable CPU (every CPU from the last decade).

`systeminfo | Select-String "Virtualization"` should show
`Virtualization Enabled In Firmware: Yes`. The line
"A hypervisor has been detected. Features required for Hyper-V will not be
displayed." is **good** — it means VBS or another hypervisor is already
running, so the platform is ready.
